1단계: 펌웨어 전처리 및 컨텍스트 추출 파이프라인 구축

LLM에게 바이너리 파일 전체를 그대로 던져줄 수는 없으므로, LLM이 이해할 수 있는 텍스트 형태의 '컨텍스트'를 자동으로 추출하는 과정이 필요합니다.

    자동화 스크립트 작성: binwalk를 이용한 파일 시스템 및 아키텍처 식별, 엔트로피 분석을 통한 암호화/압축 여부 판단을 자동화합니다.

    Headless 리버스 엔지니어링 연동: Ghidra나 IDA Pro의 Headless 모드(GUI 없이 백그라운드 스크립트로 실행)를 활용합니다.

    데이터 추출: 펌웨어의 엔트리 포인트(Entry Point), 인터럽트 벡터 테이블, 문자열(String), 사용하는 특정 메모리 주소(MMIO 추정 영역), 그리고 초기화 함수(예: Reset Handler)의 디컴파일된 C/의사 코드(Pseudocode)를 JSON 형태로 추출합니다.

2단계: RAG 및 LLM 추론 엔진 설계

제공된 기기 정보와 1단계에서 추출된 데이터를 바탕으로 LLM이 주변기기를 추론하게 만듭니다.

    데이터베이스(RAG) 구축: 분석 타겟이 되는 MCU(예: STM32 시리즈 등)의 데이터시트, 메모리 맵 레퍼런스, 레지스터 정보를 벡터 DB에 구축하여 LLM이 정확한 하드웨어 정보를 참조(RAG)할 수 있게 합니다.

    프롬프트 엔지니어링:

        역할 부여: "너는 임베디드 리버스 엔지니어다. 다음 디컴파일 코드를 보고 어떤 종류의 주변기기(UART, SPI, Timer 등)를 초기화하는지 식별하라."

        출력 규격화: LLM의 응답을 파싱하기 쉽도록 반드시 정형화된 JSON이나 YAML 포맷(예: {"mmio_address": "0x40011000", "type": "UART", "action": "dummy_read", "value": "0x20"})으로만 답변하도록 강제합니다.

    Model Context Protocol (MCP) 활용 고려: LLM이 로컬 분석 도구(Ghidra 스크립트, 파일 시스템 읽기)에 직접 접근하여 필요한 정보를 스스로 가져오도록 MCP 구조를 도입하면 에이전트의 자율성을 크게 높일 수 있습니다.

3단계: 동적 에뮬레이션 및 피드백 루프 (가장 핵심적인 단계)

LLM이 완벽한 설정을 한 번에 만들어내기는 불가능에 가깝습니다. 따라서 에뮬레이터가 멈췄을 때 이를 스스로 해결하는 피드백 루프를 구현해야 합니다.

    에뮬레이터 래퍼(Wrapper) 개발: Qiling Framework나 파이썬 기반의 QEMU 래퍼를 사용하여, LLM이 생성한 JSON 설정 파일을 에뮬레이터의 매핑 정보로 변환하고 실행합니다.

    크래시 로그 수집: 펌웨어가 매핑되지 않은 메모리(Unmapped MMIO)에 접근하여 Boot Crash가 발생하면, 해당 시점의 프로그램 카운터(PC), 레지스터 상태, 예외가 발생한 메모리 주소와 읽기/쓰기 여부를 캡처합니다.

    LLM에게 피드백 제공: "에뮬레이션 중 0x40021018 주소에 Write 접근을 시도하다 멈췄다. 주변의 코드는 다음과 같다. 이 주소가 어떤 레지스터인지 추론하고 우회할 수 있는 Dummy 값을 제안하라." 형태로 LLM에게 재질의하여 설정을 업데이트합니다.

4단계: 컨테이너화 및 검증 시스템 구성

개발된 툴체인이 다양한 환경에서 안정적으로 동작하도록 패키징합니다.

    Docker 기반 환경 분리: 복잡한 의존성(QEMU, 크로스 컴파일러, 분석 도구)을 Docker 컨테이너로 묶어 일관된 환경을 제공합니다.

    테스트 케이스 구축: 잘 알려진 취약한 IoT 펌웨어나 기본 분석용 MCU 펌웨어 샘플들을 구축하여, 자동화 툴이 어디까지 부팅을 성공시키는지 성공률을 측정하는 CI/CD 파이프라인을 구성합니다.
